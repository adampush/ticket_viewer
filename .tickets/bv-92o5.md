# Add tracker detection and datasource selection for tk

## Context / Why

Current source discovery is Beads-focused. We need deterministic tracker detection so `bv` can load from `.tickets/` when present while preserving Beads behavior.

## Scope

### In Scope

- Introduce tracker mode detection in datasource path.
- Add source discovery for `.tickets/*.md`.
- Add source type/reasoning updates for selection logs.
- Enforce deterministic precedence: non-empty `.tickets/` wins over Beads in MVP.
- Support practical discovery roots: repo root, parent-walk behavior, and `TICKETS_DIR` env var compatibility.

### Out of Scope

- Full ticket parsing/mapping logic (handled in `bv-v3f4`).
- Robot output command changes (handled in `bv-c56w`).
- Correlation/history behavior changes.

## Assumptions

- Repository root context is available where datasource discovery runs.
- Existing source selection logic can be extended without behavior regressions.

## Edge Cases to Cover

- Running `bv` from a nested subdirectory (not repo root).
- `TICKETS_DIR` set to a custom location.
- `.tickets/` exists but has only non-ticket files.
- `.tickets/` exists but all discovered ticket files are malformed.
- No `.tickets/` and no Beads sources available (clear not-found error).

## Open Questions

- How should mixed-mode diagnostics be surfaced in robot payloads while keeping schema compatibility?
  - Owner: maintainer
  - Timing: before merge of this ticket
  - Blocking: non-blocking (default: include concise diagnostic note without breaking existing fields)

## Implementation Spec

- Update datasource types and discovery to include `tk` source candidates.
- Remove hard dependency on `.beads` path acquisition in smart-load entrypoints so `tk`-only repos can load without Beads paths.
- Add validation rules for `tk` sources (directory/file existence + readable files).
- Ensure selection reason strings include `tk` source explanation.
- Capture mixed-source condition (`.tickets` + `.beads`) so downstream robot output can emit deterministic diagnostics.
- Add a non-breaking metadata path (for example a detailed load result) that exposes selected tracker/source and mixed-source diagnostics without forcing schema changes in existing robot payloads.
- Add tests for:
  - `.tickets` only
  - `.beads` only
  - mixed `.tickets` + `.beads`
  - `.tickets` present but empty
  - invocation from nested subdirectory
  - `TICKETS_DIR` override behavior
  - `tk`-only repo with no `.beads` directory through `datasource.LoadIssues(...)`
  - parent-walk discovery from nested directories (matching `tk` UX expectations)

Likely files:

## Acceptance Criteria

- Discovery identifies valid `tk` source(s) when `.tickets` has ticket files.
- `datasource.LoadIssues(...)` succeeds in a `tk`-only repo with no `.beads` directory.
- Mixed repo chooses `tk` source deterministically.
- Empty `.tickets` falls back to Beads path.
- Mixed repo detection metadata/diagnostic signal is available for output layers.
- If `tk` mode is selected but all `tk` tickets are malformed, loader returns a clear error (no silent fallback to Beads).
- If neither tracker source exists, loader returns clear actionable not-found error.
- Existing callers using `datasource.LoadIssues(...)` remain source-compatible.
- Existing Beads-only tests still pass for touched datasource packages.
